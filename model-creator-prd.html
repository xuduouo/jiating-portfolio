一、为什么做这个产品

2024 年底到 2025 年，AI 视频生成技术开始进入实用阶段——可灵、Runway、Seedance、Sora 陆续商用，单段视频质量已经可以用到实际制作中。但我在跟进这些工具时发现一个断层：

核心洞察

AI 视频工具在「单段生成」层面已经很成熟，但「从剧本到成片的完整工作流」几乎需要纯手工串联。创作者要在剧本文档、角色参考图、分镜表、AI 工具之间来回切换，大量时间花在流程管理上，而不是创作本身。

这个断层意味着一个产品机会：不是再做一个 AI 生成引擎，而是做一个「影视前期制作的 AI 工作流操作系统」，把剧本、角色、场景、分镜、生成五个环节打通，让每一步的产出自动流入下一步。

二、目标用户与场景

经过前期调研，明确了三类核心用户：

🎬 短剧创作者

有剧本但没有制作团队的个人或小团队。传统方式拍短剧成本高周期长，他们需要 AI 工具帮他们把剧本快速转化为视觉素材，效率是第一诉求。

📺 动漫番剧 Up 主

做同人动画、原创动漫内容的创作者。他们需要角色一致性保证、场景批量化生成、多镜头叙事能力——而不是一帧一帧手绘。

🏢 影视前期工作室

专业的分镜师、美术导演团队，用 AI 做概念设计预览（Previs）、风格探索和多方案快速迭代。对他们来说，工具的稳定性和输出规范比生成速度更重要。

三、核心痛点

调研过程中反复出现的几个问题：

  • 角色不一致 — AI 生成同一角色在不同镜头/场景中长相不一样，无法用于叙事后，虽然有 Seedance 等工具但操作门槛高，需要手动拼参数
  • 流程碎片化 — 写剧本用一个工具，设计角色用另一个，生成视频又换一个，中间靠复制粘贴和手动整理
  • 批量生产靠体力 — 短剧需要几十甚至上百个分镜，逐个生成、逐个下载、逐个拼接，时间全耗在重复操作上
  • AI 能力门槛高 — 要写好提示词需要懂参数、懂模型特性，不是创作者擅长的

四、产品设计原则

基于上述痛点，定下了几条贯穿始终的设计原则：

原则 01

流程驱动，而非工具驱动。 用户打开产品时看到的是创作流程（剧本→角色→场景→分镜→生成），而不是一堆 AI 模型按钮。每一步的产出自动进入下一步。

原则 02

角色一致性是底线。 如果 AI 视频连角色都认不出来，叙事就不成立。6 层身份锚点系统、角色圣经、参考图绑定——这些都是为了守住这个底线。

原则 03

批量化是第一性。 单个分镜做得再精美，如果一次只能做一个，那对于短剧这类需要 50+ 分镜的场景就没有意义。所以从架构上支持任务队列、并行执行、自动重试。

原则 04

降低 AI 使用门槛。 用户不需要懂提示词怎么写、参数怎么调，产品层面把 AI 能力封装成「写剧本→点生成→得结果」的简单交互。

五、核心链路

产品的主流程是一条线性 Pipeline：

📝 剧本 → 🎭 角色 → 🌄 场景 → 🎬 导演 → ⭐ 生成

每个环节的输出自动成为下一环节的输入。比如剧本解析后自动识别出有哪些角色、有几个场景，角色设计完成后参考图自动绑定到后续的分镜和生成环节。

同时支持非线性的工作方式——用户可以从任意一环开始，也可以跳过后面的环节只用到某一环。

六、五大板块拆解

📝 剧本解析引擎

用户输入剧本（纯文本或文件），AI 自动拆解为场景列表、分镜列表、对白文本。核心设计思路是「保留创作者的创作习惯，不做格式约束」——用户用自己习惯的方式写剧本就行，AI 来做结构化解析。

🎭 角色一致性系统

这是整个产品最核心也最难的部分。AI 生图/生视频最大的痛点之一就是角色不一致——同一个角色在镜头 A 和镜头 B 中长相不一样，叙事的可信度就破了。为此设计了一套三层递进的保障体系：

第一层：角色圣经（Character Bible）

每一个角色都有一个结构化的「圣经」文档，作为所有 AI 生成环节的权威参照。圣经不是一句简单的 prompt，而是一个包含 6 层身份锚点的体系：

📍 6 层身份锚点

① 外貌特征（Physical Base） — 脸型、肤色、发型发色、瞳色、五官比例、身高等基础生物特征。这一层决定了「这是谁」，是角色识别的底层。

② 服装细节（Costume） — 每套服装的款式、颜色、材质、配饰。支持多套服装切换（日常装/正装/战斗服），每套独立标注。这一层决定了「他今天穿什么」。

③ 气质风格（Vibe & Aura） — 角色的整体视觉气质：冷峻/温柔/凌厉/慵懒、光影偏好、色调倾向。这一层影响 AI 的氛围渲染和光线处理。

④ 标志性动作（Signature Pose） — 角色常做的标志性动作和姿态，比如抱臂站立、歪头笑、撩头发、双手插兜。AI 在生成时会参考这些动作特征。

⑤ 表情习惯（Expression Range） — 角色的表情模式：微笑时嘴角的弧度、皱眉的幅度、愤怒时的眼神方向、流泪时的面部肌肉走向。确保情感表达的一致性。

⑥ 参考图（Reference Images） — 3-5 张角色参考图，覆盖正面/侧面/半身/全身/表情特写。这些图作为 AI 的视觉锚点，是所有层级中最具约束力的一层。

设计这 6 层的核心思路是：不依赖单一维度的控制。如果只给一张参考图，AI 很容易跑偏；但如果同时约束外貌 + 服装 + 气质 + 动作 + 表情 + 参考图，六个维度互相锁定，角色漂移的概率就会大幅降低。

第二层：跨分镜自动绑定

角色圣经设计完成后，系统会自动扫描整个剧本，识别该角色出现的所有场景和分镜，自动将角色绑定到每个相关分镜的生成参数中。用户不需要在每一个分镜里重新选择角色、重新上传参考图——一次设定，全程生效。

如果用户在后续修改了角色的某个锚点（比如换了衣服），所有绑定了这个角色的分镜会自动更新，不需要逐个手动调整。

第三层：生成时的多锚点融合

当 AI 真正开始生成某个分镜时，系统不是简单地把角色圣经丢给模型，而是做一次「编译」：

  • 从角色圣经中提取与当前分镜最相关的锚点（特写镜头重点用外貌 + 表情，全景镜头重点用服装 + 姿态）
  • 将 6 层锚点融合为一条结构化的生成提示词，按优先级排序
  • 参考图作为 ControlNet / IP-Adapter 的输入，从视觉层锁定角色特征
  • 生成结果回写入角色圣经，形成持续优化的闭环

这套机制的效果是：即使使用同一个角色生成 50 个不同分镜，角色也能保持稳定的视觉一致性，不需要逐个手工修正。

🌄 场景生成

场景描述到视觉提示词的自动转换。关键决策是支持多视角联合生成——同一场景的不同角度一次性生成，保证空间一致性。

🎞️ 专业分镜系统

设计时参考了电影工业的分镜标准。支持设置景别（远景/中景/近景/特写）、机位角度、运动方式。视觉风格支持一键切换。这个板块的定位是连接「创作端」和「生成端」的桥梁。

⭐ Seedance 2.0 多模态生成

集成了最前沿的多模态生成能力，支持图片/视频/音频混合引用。设计上把复杂参数封装在后台，用户只需要告诉产品「要什么效果」。包括自动三层提示词融合、首帧图网格拼接、参数约束自动校验。

七、关键决策与取舍

做产品的过程就是不断做选择题。以下是我在 Model Creator 中最重要的几个决策，以及当时的判断逻辑：

✅ 选 Electron 而非 Web

这是最早期也最关键的架构决策。选 Web 的话更新方便、安装成本低，但有两个致命问题：一是大量图片和视频素材需要频繁读写本地文件系统，浏览器沙箱会处处掣肘；二是目标用户中有大量非技术背景的创作者，他们更习惯「下载一个软件双击打开」而不是「配置环境、装依赖、跑命令行」。Electron 虽然安装包大了些（150MB+），更新也不如 Web 即时，但换来的是完整的本地文件访问能力、离线可用、以及和传统软件一致的分发体验。从结果看，这个选择是对的——用户反馈中「下载就能用」是出现频率最高的正面评价之一。

✅ 先做流程再做单点

MVP 阶段最大的诱惑是把某个板块做深——比如花两个月把分镜系统做到媲美专业分镜软件。但我坚持先跑通「剧本→生成」的全流程，每个板块只做 60%。理由是：用户的核心痛点是「流程断了」，不是「分镜不够专业」。一个 60 分的全流程产品创造的价值，远大于一个 100 分的单点工具。事实也印证了这一点——第一版上线后用户最兴奋的反应不是「分镜功能好强」，而是「我从写剧本到出视频，中间只点了几个按钮」。等到流程跑通后，再根据用户实际使用的数据，把高频板块从 60 分往 80 分迭代。

✅ 支持多供应商

只接一家 AI 模型是最省事的做法，但也最危险。因为 AI 模型行业的变化太快了——今天最好的图生视频模型，三个月后可能就被超越了。我的判断是：产品层的价值在于「工作流编排」和「一致性保障」，而不在于绑定某个特定模型。所以从一开始就把 AI 供应商设计成可插拔的，统一抽象层屏蔽各路 API 的差异，用户可以在设置面板自由切换模型。这个决策后来被证明是明智的——Seedance 2.0 出来后我们很快就接入了，用户不需要迁移，也不需要重新学习。而且多供应商带来的另一个好处是 API Key 轮询负载均衡，一定程度上缓解了单家限速的问题。

✅ 优先级：角色一致性 > 生成质量 > 生成速度

这个优先级排序贯穿了所有功能决策。如果角色在镜头间长相不一样，单帧画面再精美也没有叙事价值。所以我们宁愿在角色锚点体系上多花功夫（6 层身份锚点、角色圣经、参考图绑定），哪怕这意味着每段生成的前置准备时间会长一些。生成质量排在第二——在保证角色一致的前提下，尽可能选择画质最好的模型和参数。生成速度排最后——用户对「等 5 分钟还是 10 分钟」的敏感度，远低于对「角色像不像」的敏感度。

✅ 本地优先，云端为辅

考虑到用户处理的是大量高清图片和视频素材，全部走云端既不现实（上传下载耗时长），也有隐私顾虑。所以核心数据（角色圣经、剧本、分镜表、参考图）都存在本地，AI 生成任务通过 API 发送到云端执行。这样既保证了日常操作的流畅性，又能利用云端算力做大规模生成。本地存储的另一个好处是离线可用——用户可以在没有网络的环境下写剧本、设计角色，联网后一键提交生成。

❌ 不做视频编辑

最艰难的拒绝。很多用户问「能不能在软件里直接剪辑」，市场上也有足够的理由做——一站式体验、减少工具切换。但最终决定不做，理由有三：一是视频编辑是成熟赛道，剪映、PR 已经做到极致，我们不可能做得更好，也不应该去竞争；二是加入视频编辑会彻底改变产品形态——从「创作流程工具」变成「剪辑软件」，团队的精力和技术栈都会被拖进去；三是用户真实的工作流是「AI 批量生成 → 人工精剪」，而不是「AI 生成即出片」。所以我们坚持 Model Creator 的边界停在「生成完成」这一刻，输出高质量的视频片段，后续交给专业剪辑工具。与其做一个 60 分的编辑器，不如把生成环节做到 90 分。

❌ 不做移动端

MVP 阶段有不少用户问有没有手机版。没做。因为影视创作的剧本设计、角色管理、分镜编排这些操作在手机上体验很差——屏幕太小、文件管理不便、AI 生成结果预览需要大屏。我们的核心场景是「坐在电脑前批量制作」，移动端不是目标战场。这个决策也帮我们避免了在响应式设计和移动端适配上的大量投入。

八、复盘与收获

从一个想法到 3.7k+ Stars 的开源项目，中间踩过坑也做过对的选择。以下是这个产品教会我的几件事：

💡 从用户场景出发，而不是从技术出发

很多 AI 产品是「模型出了新能力→我堆个功能」。比如 Seedance 2.0 出来了，就做一个 Seedance 2.0 的生成页面——这本质上是模型的展示台，不是产品。

Model Creator 是反过来做的：先想清楚创作者的完整工作流是什么样的——他们怎么写剧本、怎么管角色、怎么排分镜、怎么批量生成——然后再看 AI 能在这个流程的哪个环节提效。不是「AI 能做什么所以做什么」，而是「创作者需要什么所以用 AI 做什么」。这个思维方式决定了产品形态不是一堆 AI 功能的堆砌，而是一条完整的创作 Pipeline。

举个具体的例子：剧本解析功能不是因为我们能做 NLP 才做的，而是因为观察到一个真实的用户行为——创作者在微信/飞书/Word 里写好了剧本，然后手动复制到各个 AI 工具里分段生成。我们的价值不是「解析剧本的技术」，而是「让用户少复制粘贴一次」。

💡 流程感比功能列表重要

用户不会因为你有 20 个功能就觉得好用，而是因为「从我开始到我希望的结果，中间只需要很少几步」。流程设计是第一优先级。

在 MVP 阶段做了一件事：让用户从头到尾跑通一次。观察他们在哪一步卡住、在哪一步困惑、在哪一步退出。结果是——用户最常问的不是「这个功能怎么用」，而是「我现在该做什么」。这是流程设计失败的信号。所以后续把所有操作都组织成线性的五步（剧本→角色→场景→分镜→生成），每一步做完自动提示下一步去哪，每一步的产出自动流入下一步。用户不需要理解软件的结构，只需要跟着走。

从数据上看也有印证：加了流程引导之后，新用户从打开软件到第一次成功生成视频的完成率提高了约 40%。用户不需要学习「软件的功能」，只需要理解「自己现在在哪一步」。

💡 克制地定义产品边界

最大的诱惑是不断加功能。明确不做视频编辑、不做音频处理、不做平台型生态——这些决策可能比做什么更重要。

产品经理最容易犯的错误是「用户要什么就给什么」。用户说想要剪辑功能，就加剪辑；用户说想要音效库，就加音效；用户说想要模板市场，就加市场。但产品的本质不是需求收集器——好的产品是一个有明确边界的系统，知道什么该做什么不该做。

Model Creator 的产品边界是用一句话定义的：「从剧本到生成，中间的每一步」。在这条线之内的，全力以赴；在这条线之外的，坚决不做。视频编辑在线外，音效处理在线外，发布分发在线外。这个边界帮我们在资源有限的情况下，把生成这个环节做到了足够的深度。

一个验证：Roadmap 上被拒绝的需求远比被接受的多。每一个「不做」的决策背后都对应了一个明确的判断——这个功能如果做了，会分散多少精力、会模糊多少产品定位、会让用户对「Model Creator 是什么」产生多少困惑。做减法的能力可能比做加法更难，但也更重要。

本地搭建步骤

环境要求

Node.js ≥ 18 + npm ≥ 9

# 安装依赖
npm install

# 启动开发模式
npm run dev

# 配置 API Key（在设置面板填入）
# 打包 Windows 安装程序
npm run build

关键设计特点

  • Electron 桌面应用 — 跨平台，本地文件系统访问，大数据处理。非技术用户「下载安装即可用」，无需配置环境或命令行操作。
  • React + TypeScript 前端 — 五个主面板各对应一个核心创作环节，通过 Zustand 实现多面板间的数据共享：角色信息在剧本面板→场景面板→分镜面板之间自动流转。
  • AI Core 引擎 — 封装了提示词编译、角色圣经维护、多供应商任务轮询等复杂逻辑，上层 UI 只需调用简单 API 即可触发 AI 任务。
  • Seedance 2.0 集成 — 最新的多模态视频生成技术，支持图像/视频/音频混合引用，通过统一抽象层屏蔽各供应商 API 差异。
  • 本地优先，云端为辅 — 核心数据（角色圣经、剧本、分镜表、参考图）存储在本地，AI 生成通过 API 发送到云端执行，支持离线编辑、联网后一键提交。

本地源码在 D:\moyin-creator\，npm run dev 即可启动。
